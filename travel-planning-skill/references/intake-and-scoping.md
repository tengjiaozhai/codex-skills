# Intake And Scoping

## Required Slots

Before doing serious itinerary work, try to fill these slots:

| Slot | Required when |
| --- | --- |
| Destination | Always |
| Departure city | When transport comparison matters |
| Start date | Always |
| End date | Usually; if missing, ask total days |
| Total days | When not implied by dates |
| Budget | Always for meaningful planning |
| Travelers | Ask when it affects room type, cost, or rhythm |
| Must-go / must-skip | Ask when user has preferences or too many sights |

## Mandatory Clarification Rule

If the request is under-specified, ask concise questions covering:

- `去哪儿玩 / 从哪儿出发`
- `哪天到哪天玩`
- `玩几天`
- `预算多少`

If the user already gave one or more of these, ask only for the missing ones.

## Compact Question Style

Prefer one short bundle instead of many tiny follow-ups.

Good pattern:

1. 去哪里玩、从哪里出发？
2. 哪天到哪天玩，或者一共玩几天？
3. 总预算大概多少？

Optional add-ons when useful:

4. 几个人出行，情侣 / 家庭 / 朋友？
5. 有没有必须去或不想去的景点？

## Date Consistency

If the user gives both dates and day-count, validate them.

Example mismatch handling:

- “你是按 5 月 1 日到 5 月 4 日的 4 天 3 夜来排，还是按 5 天游来排？”

In final output, always use absolute dates.

## Safe Defaults

Use defaults only when the user clearly invites you to choose.

Reasonable defaults:

- pace: relaxed, not extreme special-forces mode
- budgeting style: hard costs first, then optional upgrades
- output style: recommendation + daily plan + budget

Do not silently assume:

- departure city
- trip dates
- total budget
- traveler count

## Ambiguous Destination Handling

If destination itself is vague, stop and clarify before searching.

Examples:

- “五一想出去玩” -> ask destination + dates + budget
- “帮我安排个海边” -> ask origin, dates, budget, and preferred region
- “周末玩台州还是温州” -> compare both only after dates and budget are known
